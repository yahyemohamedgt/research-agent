# Every array-of-objects field in the synthesis tool schema needs explicit properties

`winning_ads` and `platform_breakdown` were originally declared as `{"type": "array", "items": {"type": "object"}}` — a bare object with no `properties` or `required` list. This gave Sonnet no contract for what keys to use, and it invented its own (`angle`, `format`, `notable` instead of `brand`, `hook_text`, `emotional_driver`, `days_running`, `url`) as soon as `winning_ads` actually had real data to populate — which only started happening after the Foreplay query fix (ADR 0009). Before that, `winning_ads` was always an empty list, so the schema gap was invisible: an empty array has no keys to get wrong. Both the web UI and CLI render code expect the exact keys defined in `state.py`'s `WinningAd`/`PlatformBreakdown` TypedDicts, so the mismatch rendered as blank Hook/Emotion/days fields with no error anywhere. The fix adds explicit `properties` and `required` to both schemas, matching the TypedDicts field-for-field. The lesson generalizes: any `{"type": "object"}` with no `properties` in a forced-tool-use schema is a landmine that only detonates once the model actually has content to put there — worth auditing for on any future field added the same way.
